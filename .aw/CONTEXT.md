# Context

## Domain Language

| Term | Definition | Avoid |
| --- | --- | --- |
| **Skill** | A slash-command script installed into a CLI tool's skills directory; the app ships a bundled set and can sync them to each configured CLI. | Plugin, command |
| **Skill Target** | The specific CLI ecosystem (claude, codex, pi, openCode) and its corresponding directory where skills are installed. | CLI target |
| **Required Skills** | The ordered list of skills the app checks for presence and manages in the Skills Preferences pane. | Bundled skills |
| **SkillClassifier.State** | The status of a skill on disk: `missing` (not installed), `clean` (matches bundle), `stale` (matches a prior bundle hash), or `modified` (locally edited). | |
| **Workflow** | The fixed Ralph sequence: Plan (grill-with-docs → to-prd → to-tasks) → Build (ralph) → Verify (qa). | Pipeline |

## Relationships

- A **Skill Target** maps one-to-one to a CLI preset (claude → `~/.claude/skills/`, openCode → `~/.config/opencode/commands/`, pi → `~/.agents/skills/`).
- The **Skills Preferences Pane** shows one `DirectorySection` per active Skill Target, each listing all Required Skills with their classification state.

## Decisions

- `PresenceChecker.requiredSkills` drives both the prerequisite check and the Skills pane row order.

## Decisions

- Skills are portable across Claude Code, Codex, and Pi (same `SKILL.md` format). OpenCode uses a flat `.md` format but is handled separately. All four targets should show and allow installation of all required skills.
- The per-row action button is labeled **"Install"** when a skill is `missing` and **"Update"** when it is `clean`, `stale`, or `modified`.
- `requiredSkills` order matches workflow execution order: `excavation-agent, grill-with-docs, to-prd, to-tasks, ralph, qa`.
- Inline rename commits (saves current text) when focus leaves the sidebar — e.g., clicking into the terminal. It does not cancel.

## Feature: Architecture Workspace

### Domain Language

| Term | Definition | Avoid |
| --- | --- | --- |
| **Architecture Workspace** | A Plan-phase feature where the developer selects a codebase entry point, an AI agent excavates the relevant component neighborhood, and the developer inspects and arranges the inventory before implementation begins. | |
| **Inventory Item** | A discovered component, pattern, service, or relationship surfaced by the AI during excavation. Outskirts nodes contain: name, file path, AI-inferred properties/methods, and a one-line role description (shown on hover). Inskirts nodes: name, pattern type, developer-defined properties/methods, and a one-line role description (shown on hover). | |
| **Excavation Loop** | The iterative cycle where the developer asks a natural-language question, the AI digs deeper into the codebase, and newly discovered pieces are added to the Inventory. | |
| **Workbench** | The full tab in SessionDetailView containing the Canvas, PatternPalette, and ExcavationChatView. The container for the spatial construction experience. Active throughout the Plan phase. | Architecture Workspace (when referring to the spatial UI), Workspace |
| **Braindump** | The developer's unstructured natural-language description of the feature they want to build. Stored as `braindump.md`. Seeds the Excavation Agent and provides starting context for grill-with-docs. Already exists in the app; the Workbench extends its role. | seedIdea, Brainstorm Prompt |
| **Excavation Agent** | The AI agent that runs immediately after the Braindump is submitted, scanning the codebase autonomously to populate the initial Outskirts inventory before grilling begins. | |
| **Canvas Serialization** | An LLM-readable text representation of the current Workbench state — node names, types, roles, and connections — passed to the grill AI so it can reason about what the developer is building without reading raw positional/UI data. | |
| **Outskirts** | Existing codebase components surfaced by the AI — read-only anchors at the periphery of the Workbench. The developer connects to these but does not modify them. | |
| **Inskirts** | The developer's construction zone at the center of the Workbench. New components, interfaces, and relationships are placed here during feature architecture. | |
| **Connections** | Typed bridges the developer draws between nodes. Type is picked from a fixed set: `owns`, `delegates to`, `observes`, `calls`, `implements`, `adapts`. Type is suggested via rule-based lookup on pattern types (e.g., `* → Protocol = implements`); a stubbed AI call interface exists for post-MVP enhancement. | |
| **Pin** | A public method or property on a node — the conceptual connection point between components. Pins are rendered on the node card (MVP). Helper/private methods are internal and not shown. | |
| **Beach Blanket** | A cosmetic grouping rectangle on the canvas that labels a subsystem containing multiple nodes. No connection semantics — purely visual orientation. | Module (when used to mean a grouping) |
| **Module** | A conceptual unit of encapsulation from the `/to-prd` skill — a group of related responsibilities with a simple, testable interface. Maps roughly (but not always exactly 1:1) to a named type in the target language. | Class, component |
| **Canvas** | The interactive node graph within the Workbench tab — the visual surface where Outskirts and Inskirts nodes are placed and Connections are drawn. Distinct from the Workbench (the full tab) and from Architecture Workspace (the feature). | Workspace (when referring to the node graph) |
| **Symbol Index** | The compact repository-wide code index produced by `SymbolExtractor`: all named types in the repo with their file paths and public members. Shared read-only context for planning agents, including ExcavationAgent and grill-with-docs. | |

### Decisions

- Architecture Workspace is a new feature inside AgentWorkflows, placed in the Plan phase of the Workflow.
- The Workbench is **ephemeral and per-feature** — opened for a feature, used during construction, closed when shipped. No maintenance burden; the value is in the construction process, not the artifact.
- Architecture Workspace is a **persistent pane**, not a discrete skill. It stays active across the entire Plan phase and accumulates Inventory Items from all Plan-phase sources. (ADR 0001)
- **Plan phase sequence:** Braindump → Excavation Agent populates Outskirts → grill-with-docs runs with Workbench active alongside; developer constructs Inskirts **during** grilling → to-prd → ARCHITECTURE.toml. (ADR 0002)
- `seedIdea` in `SessionDetailView` should be renamed to `braindump` to unify with the file name and engine conventions.
- Canvas state is persisted to `canvas.toml` in the session directory. CLI processes (Excavation Agent, grill-with-docs) read and write it; the app watches it via `DirectoryWatcher`. No direct process injection — all cross-boundary communication is file-mediated. (ADR 0003)
- The `Workbench` tab is always-visible and always-editable across all Workflow phases — no `Workflow` model changes needed. Construction typically happens during the Plan phase but the canvas can be edited at any time.
- `ARCHITECTURE.toml` and `canvas.toml` live in the session directory, managed via `SessionDirectoryLayout`.
- **ExcavationAgent is a skill** (like grill-with-docs), not a Swift module. It runs in the ExcavationChatView terminal automatically at session start. Before launching it, the app ensures a repository-wide `symbol-index.toml` exists and is current. The skill reads that shared index plus `braindump.md`, selects 10–15 relevant Outskirts nodes, and writes them to `canvas.toml`. App owns all extraction; the skill never touches `SymbolExtractor` directly. (ADR 0006, ADR 0010)
- **`SymbolExtractor` replaces ast-grep.** A Swift module that produces `symbol-index.toml` in-process. Swift files use SwiftSyntax (Apple's canonical parser — added as a Swift Package dependency) via an AST visitor that collects `ClassDeclSyntax`, `StructDeclSyntax`, `EnumDeclSyntax`, `ProtocolDeclSyntax` and their public members. Objective-C, JavaScript, Python, and GDScript use lightweight regex patterns to extract named types and public members. No subprocess, no binary to codesign. Unsupported or unrecognized languages surface a warning and the canvas starts empty (excavation continues via chat).
- **`symbol-index.toml` format:** `[[types]]` array-of-tables. Each entry: `name` (type name), `file` (repo-relative path), `members` (array of strings in pin format: `"name: Type"` for properties, `"method() -> Return"` for methods). TOML chosen over Markdown for consistency with `canvas.toml` and explicit parse-error detection. `members` maps directly to `pins` in `canvas.toml` — no transformation needed when the skill writes selected entries as `[[outskirts]]`.
- **`symbol-index.toml` is repository-wide, not session-scoped.** New sessions reuse the same index instead of rebuilding from scratch. Session artifacts such as `canvas.toml`, `canvas-layout.toml`, and `ARCHITECTURE.toml` remain per-session. (ADR 0010)
- **`excavate` is a new WorkflowEngine step type.** When executed: (1) runs bundled ast-grep to produce `symbol-index.md`, (2) dispatches ExcavationAgent skill prompt to the excavation engine. Visible in WorkflowInspector; replayable. Requires EngineManager to track two AgentEngine instances per session. (ADR 0008)
- **EngineManager keys engines by `(sessionID: UUID, role: EngineRole)`**, where `EngineRole` is an enum with at minimum `.main` and `.excavation` cases. All existing call sites map to `.main`. Extensible if a third engine role is added. (ADR 0009)
- **Excavation CLI** is a dedicated section in the preferences pane (alongside existing Skill Target sections) for configuring which CLI target runs the ExcavationAgent skill. Independent from the main workflow CLI target.
- After ExcavationAgent completes, the developer continues in the same ExcavationChatView terminal for freeform follow-up excavation questions.
- **Re-excavation merges, not replaces.** When the `excavate` step is re-run from WorkflowInspector, the ExcavationAgent writes only new `[[outskirts]]` entries not already present in `canvas.toml` (matched by name). Existing Inskirts nodes, connections, Beach Blankets, and already-present Outskirts nodes are untouched. There is no wipe-and-restart path — if the developer wants a clean canvas they delete nodes manually.
- grill-with-docs runs separately in the main terminal, distinct from ExcavationChatView. Both write to `canvas.toml`.
- The Excavation Agent is a prompt/process invocation that runs before grill-with-docs (seeded by the braindump), similar to how grill-with-docs is invoked. grill-with-docs is updated to read `canvas.toml` for context and write new Outskirts discoveries to it during the interview.
- When Claude discovers something worth surfacing, it **auto-adds** it to the Inventory without waiting for developer action — analogous to how ADRs are auto-written. (ADR 0001)
- **Node granularity:** one node = one named type (class, struct, enum, or equivalent in the target language). Public methods and properties are Pins, shown on the node card. Helper/private methods are invisible to the canvas.
- **Connections are node-to-node** for MVP. Pin-to-pin precision (e.g., "WorkbenchModel.serialize() calls CanvasFileStore.write()") is deferred post-MVP.
- The canvas is **curated, not exhaustive** — only components architecturally relevant to the feature under construction appear. A 5000-line class is still one node.
- **Excavation is two-phase:** Phase 1 — `SymbolExtractor` builds the Symbol Index deterministically; Phase 2 — AI selects 10–15 relevant nodes from the index and writes them to `canvas.toml`. The same Symbol Index is also available to other planning agents that need bounded codebase context. (ADR 0006, ADR 0010)
- **App owns all ast-grep invocations.** Binary is bundled with the app. grill-with-docs writes name + file path only; the app enriches new entries with pins via ast-grep. (ADR 0006)
- **Supported languages (MVP):** Swift, Objective-C, JavaScript, Python, GDScript. Unsupported languages show an explicit warning; canvas starts empty and developer excavates via chat only.
- **`canvas.toml` format:** TOML. Sections: `[[outskirts]]` (name, file, role, pins), `[[inskirts]]` (name, pattern, role, pins), `[[beach_blankets]]` (name, nodes list), `[[connections]]` (from, to, type). `from`, `to`, and `beach_blankets.nodes` reference nodes by name; names must be unique across the canvas — CanvasFileStore fails gracefully (surfaces error, skips duplicate) on collision. LLMs append new `[[outskirts]]` blocks; app parses via TOMLKit. Chosen over JSON (too much boilerplate), Markdown (too fragile to parse), and YAML (indentation-sensitive; LLM appends can silently corrupt the file). Node coordinates are app-internal only — never written to `canvas.toml`.
- **`excavation-agent` is a required skill** listed in `PresenceChecker.requiredSkills` and shown in the Skills preferences pane with the same Install/Update row behavior as other skills. Execution order: `excavation-agent, grill-with-docs, to-prd, to-tasks, ralph, qa`.
- **Planning agents start from `symbol-index.toml`, not repo-wide search.** The skill prompts instruct agents to use the pre-built Symbol Index for initial context and inspect only specific files as needed, rather than invoking Bash, grep, or file-tree scans across the whole repo. The app owns all extraction.
- **`grill-with-docs` skill modifications:** reads `canvas.toml` at the start of each question loop for architectural context; when it discovers a relevant existing component, appends a bare `[[outskirts]]` block (name + file only, no pins) to `canvas.toml`. Append semantics: the skill must re-read `canvas.toml` immediately before writing — never use a version read earlier in the loop — to avoid overwriting concurrent entries from other writers. The canvas.toml schema is not inlined in SKILL.md; a lean instruction points at a `references/` file per skill-creator guidance.
- **Pin enrichment lives in `CanvasFileStore`.** When `DirectoryWatcher` fires and `CanvasFileStore` parses the updated `canvas.toml`, it detects any `[[outskirts]]` entry with a missing or empty `pins` field, runs `SymbolExtractor` on that file, and writes back the enriched entry. `WorkbenchModel` only ever receives fully-enriched entries — it has no enrichment logic.
- **ExcavationChatView is a separate agent from grill-with-docs.** grill-with-docs drives the structured planning interview. ExcavationChatView is a freeform codebase search agent — developer asks ad-hoc questions, it searches and appends findings to `canvas.toml`. They run simultaneously; both write to `canvas.toml`.
- **PatternPalette has two kinds of entries:** (1) **Multi-node blueprints** for structural patterns with well-known participants (Observer, Repository, Adapter, Factory) — dragging expands into 2–4 pre-connected nodes and auto-creates a Beach Blanket named after the pattern; (2) **Single-node archetypes** for patterns that are naturally one type (Service, Coordinator, Delegate, generic) — dragging places one node with no Beach Blanket. The `pattern` label on a node is set by the blueprint/archetype on placement and editable in the inspector. Custom labels (e.g. ViewModel, Manager, Reducer) are allowed and fall back to grey in the node color scheme. Pattern labels drive rule-based connection-type suggestions regardless of whether they match a palette entry.
- **Multi-node blueprints auto-create a Beach Blanket** named after the template (e.g. "Repository Pattern"). The blanket is immediately renameable and deleteable; deleting it leaves the nodes on canvas. Single-node archetypes do not auto-create a blanket.
- **Beach Blanket creation:** Option+drag on the canvas background draws a rectangle; nodes inside the drag area become the blanket's initial members. After drag release, the blanket appears with an inline name field focused. This is the only manual creation gesture — blankets are also auto-created by multi-node blueprint templates.
- **Beach Blanket node eviction:** dragging a node outside a blanket's bounds detaches it from that blanket. The blanket shrinks to fit its remaining nodes. Dragging a node inward past a blanket's edge adds it to that blanket (auto-grow).
- **Beach Blanket membership is non-exclusive.** A node can belong to multiple blankets simultaneously (e.g., `ImportCoordinator` in both "Import Subsystem" and "Core Services"). Blankets overlap visually on the canvas. `beach_blankets.nodes` is a plain name list with no exclusivity constraint — no parent/child hierarchy.
- **"Finalize Canvas" is idempotent** — pressing it any number of times overwrites `ARCHITECTURE.toml` silently. No "already finalized" state. If the canvas has no connections yet, a soft inline warning is shown ("Canvas has no connections — ARCHITECTURE.toml will be nearly empty") but the write is not blocked.
- **Node card layout follows ComfyUI conventions.** Each card shows: pattern label (small caps, top) + node name + file path (outskirts only). Properties (non-method pins) appear as socket dots on the **left** border with inline labels; methods appear on the **right** border. Labels are always visible, not hidden behind hover. Hovering the card body shows the one-line role description as a native tooltip. Connections are node-to-node for MVP — sockets are visual indicators of the interface, not drag targets yet.
- **Connection routing uses adaptive bezier.** Control point offset scales with horizontal distance when the destination is to the right (`t = dx * 0.5`); when the destination is to the left or close, offset scales with vertical distance (`t = max(dy * 0.5, 90)`) so the curve sweeps wide rather than folding back on itself.
- **Connection color follows the source node's color** (same color as the node card accent). Default on; user-configurable. Supersedes the earlier inskirts=blue/outskirts=orange zone rule — zone color was a two-value approximation; per-node color is richer and still encodes provenance.
- **Node card color is pattern-type-based by default** (Observer=teal, Repository=purple, Service=blue, Coordinator=green, Delegate=brown, Adapter=yellow-green, Factory=rose, unknown=grey — HSB saturation 0.55, brightness 0.75 for dark-mode safety). Color algorithm is user-configurable (options: pattern type, golden-ratio hue, name hash).
- **Node card width is dynamic, no truncation.** Cards stretch from a 120pt minimum to a 300pt maximum; text wraps within that budget. No ellipsis. This preserves full pin label readability (including long method signatures) while the variable card width itself becomes a visual distinguisher. Width behavior is user-configurable.
- **Canvas visual style is dark** (ComfyUI-style: dark canvas background ~`Color(white: 0.11)`, dark card body ~`Color(white: 0.17)`, colored accent border matching the node's pattern-type color). This is fixed — not user-configurable — because the dark surface is what makes the colored wire and card accents readable.
- **Canvas grid overlay is user-togglable** (default off). When on, draws a faint dot/line grid at 40pt spacing in a slightly lighter shade of the canvas background. Setting persists in user preferences.
- **Reroute nodes are pure canvas UI — never written to `canvas.toml`.** They live in `WorkbenchModel` alongside node coordinates: an array of reroute waypoints (id, x, y). A connection may reference an ordered list of reroute IDs; the renderer chains adaptive bezier legs through them. The developer places reroutes from the canvas (e.g. Shift+click a wire) to redirect lines around obstacles. Visual: a small filled circle matching the connection's color.
- **Spatial state persists to `canvas-layout.toml`** in the session directory (sidecar alongside `canvas.toml`). Stores node coordinates (UUID → x, y), zoom level, pan offset, and reroute waypoints. UUIDs are the stable identity key used only in `canvas-layout.toml` — they never appear in `canvas.toml`. Written by the app on every spatial change; read on session open. On first open (no sidecar exists), the app applies auto-layout: Outskirts nodes placed around the periphery, Inskirts nodes clustered in the center. `canvas-layout.toml` is managed via `SessionDirectoryLayout` alongside `canvas.toml` and `ARCHITECTURE.toml`.
- **`canvas.toml` uses names as reference keys throughout** — connections, Beach Blanket node lists, and all cross-references use the human-readable `name` string. This keeps the file readable by both the developer and LLMs. On node rename, `CanvasFileStore` performs a single atomic rewrite of `canvas.toml` updating the `[[inskirts]]` name field, all `[[connections]]` `from`/`to` fields, and all `[[beach_blankets]]` `nodes` lists in one pass. The existing re-read-before-write rule makes stale-name collisions negligible.
- **Connection selection and editing:** clicking a wire (within a ~10pt invisible hit area around the stroke) selects it — wire highlights (thickens slightly) — and opens it in the right inspector panel showing `from`, `to`, and `type`. Type is changed via a picker in the inspector. Delete/Backspace removes the connection. Right-click on a wire shows a context menu with "Delete Connection" as the primary action for discoverability.
- **Default connection interaction is click-to-arm.** Click an output pin to arm it (pin fills + scales); the pending wire follows the cursor. Click an input pin to complete the connection. Click the armed pin again or tap empty canvas to cancel. Drag-to-connect (hold + drag from output pin, release on input pin) is a user-configurable settings option.
- **Wire snap uses closest-pin-wins within a ~20pt radius.** When the pending wire endpoint enters the snap radius, it locks to the nearest input pin. The pin glows and scales up as snap feedback. All major node editors (Blueprint, ComfyUI, Blender) use closest-pin as the resolution rule; the standard radius is ~1–1.5× pin diameter. Post-MVP: add type filtering (dim incompatible pins during drag) to eliminate snap ambiguity without needing a larger radius.
- **Wire layer renders above node cards** so connections visually plug into pins rather than passing under cards. Connected pins fill solid: output pins fill with the node's own pattern color; input pins fill with the source node's color (traceable at a glance without following the wire).
- **Corrupted `canvas.toml` and orphaned references both use best-effort parse with a single warning banner.** CanvasFileStore silently drops malformed entries and connections referencing missing node names; all issues are surfaced in one inline canvas warning banner (e.g. "canvas.toml had parse errors — some nodes may be missing. Check the file manually."). No blocking error. Developer can open the file from the Files tab to inspect and fix it.
- **Empty excavation shows an inline canvas banner:** "Excavation found no relevant components — try describing the feature in more detail, or ask a question in the excavation terminal below." The canvas stays empty, the ExcavationChatView drawer stays open, and the developer can proceed to place Inskirts manually. No blocking state.
- **Role descriptions are generated only by ExcavationAgent**, which has the braindump context needed to infer a meaningful one-liner. Grill-discovered nodes arrive with no role (empty field); the developer can add one manually in the inspector. No AI role-generation call during pin enrichment — a decontextualized call would produce generic descriptions.
- **Auto-placement for AI-arrived Outskirts nodes uses a peripheral ring.** When `canvas-layout.toml` has no position for an Outskirts node (new arrival), the app places it on a ring around the existing canvas content at the least-occupied arc position. Ring radius scales with node count to prevent overlap. Batch arrivals (initial 10–15 from ExcavationAgent) are distributed evenly around the full ring. Developer can reposition nodes freely after placement.
- **`canvas.toml` is the shared contract between human and AI.** The AI reads and writes semantic graph data (nodes, edges, types, roles); the app owns all spatial/visual state (coordinates, zoom, pan). The AI never reads or writes coordinates — topology is expressed through edges, not position.
- **Canvas chrome is a slim fixed top toolbar + bottom drawer + right inspector panel.** Toolbar contains: workflow name (monospaced), zoom in/out + percentage, Run button. The **Run button re-runs the `excavate` step** — runs SymbolExtractor in-process then launches the ExcavationAgent skill in ExcavationChatView — without requiring a trip to WorkflowInspector. Uses merge semantics (same as re-excavation from WorkflowInspector). Disabled while excavation is already running.
- **Zoom and pan:** pinch-to-zoom and two-finger scroll to zoom; drag on empty canvas background or two-finger pan to pan. Zoom range 25%–400%, default 100%. Toolbar zoom in/out buttons step by 25%. Double-tap the zoom percentage label or ⌘0 for fit-to-content. The excavation agent terminal opens as a bottom drawer that slides up from the screen edge (canvas is the primary surface; terminal is secondary). A minimap pill sits in the bottom-right corner of the canvas and drifts upward when the drawer is open. A **right inspector panel** opens when a node or connection is selected; it shows all editable fields for that selection (name, pattern type, role description, pins). The inspector is the sole editing surface — no inline editing on the card itself.

### Open Questions

- ~~What does "visual workspace" mean concretely in this app?~~ → Node canvas from MVP. The Outskirts/Inskirts spatial separation *is* the feature; a list loses it. The mental model is a **circuit board**: Outskirts components are existing parts soldered at the edges (read-only, with exposed pins/interfaces); the Inskirts is the empty board space where the developer lays new traces and components; Connections are the traces drawn between pins.
- ~~What does "excavate" mean technically?~~ → Claude explores autonomously via Bash/grep/file tools (Option 3).
- The excavation is **iterative**, not one-shot: developer asks natural-language questions ("is there a way to extract all transactions cleanly?"), AI digs deeper, and newly discovered pieces surface as additional Inventory Items. This prevents missing inventory from being discovered only during later architectural feedback.
- ~~What is the primary output?~~ → Canvas and PRD are **orthogonal artifacts** that coexist. PRD = requirements, no implementation details. Canvas = architectural blueprint, implementation-focused. Both are durable references for the ralph loop alongside ADRs and CONTEXT.md. The Canvas must be serialized to an LLM-readable, efficient format as a persistent artifact.
- Canvas artifact is named **`ARCHITECTURE.toml`**. Contains only connected Outskirts nodes (existing components the feature actually integrates with), all Inskirts nodes (new components to build with pattern type and responsibilities), and their Connections. Unconnected Outskirts nodes are excluded.
- **Palette:** Hardcoded blocks organized by category (macOS, backend, etc.) for convenience only. Developer can mix and match freely — category is a filter/organizer, not a constraint. Pattern blocks: Delegate, Observer, Adapter, Factory, Repository, Coordinator, Service, Protocol (macOS). Generic blocks alongside. No session-level stack setting; palette lives in the Workbench tab.
- **UI placement:** Workbench lives as a new tab in `SessionDetailView` alongside terminal/iterations/files/diff/log. Canvas is full-width within the tab. Visible during Plan phase.
